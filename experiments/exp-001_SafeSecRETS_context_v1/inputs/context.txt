A arquitetura do sistema SafeSecRETS segue uma abordagem híbrida que combina arquitetura em camadas com arquitetura orientada a eventos (EDA), com o objetivo de garantir separação de responsabilidades, desacoplamento e suporte a atualizações em tempo real .

O sistema é dividido em quatro camadas principais. A camada de apresentação (Presentation Layer) é responsável pela interface com o usuário e foi construída utilizando a plataforma Bubble. Nela estão os componentes visuais como o SafeSecIoT Canvas, utilizado para planejamento do projeto, e o pipeline STPA-SafeSecIoT, que guia a análise de segurança e requisitos. Essa camada também contém os workflows do Bubble, que processam as ações do usuário, e um conector de APIs responsável por realizar chamadas ao backend.

A camada de comunicação (Communication Layer) atua como intermediária entre frontend e backend, sendo implementada como um plugin em JavaScript. Ela utiliza WebSockets para manter uma conexão persistente com o backend (Supabase Realtime), permitindo que eventos como inserções, atualizações e deleções no banco de dados sejam enviados em tempo real para a interface. Isso elimina a necessidade de consultas constantes ao banco e melhora o desempenho e a responsividade do sistema .

A camada de persistência (Persistence Layer) é baseada no Supabase, que oferece um banco de dados PostgreSQL, APIs REST automáticas e mecanismos de autenticação. Essa camada é responsável por armazenar os dados do sistema, controlar o acesso dos usuários e disponibilizar operações de leitura e escrita via API. Além disso, o Supabase Realtime monitora mudanças no banco e envia eventos para a camada de comunicação, permitindo atualização em tempo real da interface .

Por fim, a camada de inteligência artificial (AI Assistant Layer) integra um modelo de linguagem (LLM), especificamente o Mixtral-8x22B-Instruct, acessado via Fireworks AI. Essa camada é responsável por gerar automaticamente requisitos e restrições de segurança a partir de informações estruturadas fornecidas pelo usuário, funcionando como um serviço externo desacoplado do restante do sistema .

O fluxo geral do sistema ocorre da seguinte forma: o usuário interage com a interface na camada de apresentação, o que aciona workflows que fazem requisições à API do backend. Os dados são persistidos no banco PostgreSQL e, quando há alterações, o Supabase Realtime envia eventos via WebSocket para a camada de comunicação, que atualiza a interface em tempo real. Paralelamente, quando necessário, a camada de IA é acionada para gerar informações adicionais com base em prompts estruturados.

Essa arquitetura permite colaboração em tempo real, integração com inteligência artificial e uma separação clara entre interface, comunicação, persistência e processamento inteligente.
