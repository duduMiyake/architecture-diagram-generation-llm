O sistema descrito é estruturado como uma ferramenta de análise estática para verificação de conformidade arquitetural em projetos C#, organizada como uma biblioteca que atua sobre o código-fonte e segue um modelo arquitetural modular com responsabilidades bem definidas e fluxo de processamento sequencial. A arquitetura geral pode ser compreendida como um pipeline de análise que inicia na ingestão da solução de software e culmina na geração de relatórios de violações arquiteturais, utilizando uma API fluente como camada de abstração para interação com o usuário.
O ponto de entrada do sistema é o módulo central denominado Arch, responsável por orquestrar todo o processo. Esse módulo recebe como entrada uma instância de Solution proveniente da plataforma de compilação, que encapsula todos os projetos C# e suas dependências. A partir dessa entrada, o módulo Arch coordena a inicialização da análise e mantém uma representação centralizada dos dados arquiteturais extraídos, funcionando como um facade que abstrai a complexidade dos demais componentes internos.
A etapa seguinte é conduzida pelo módulo AST, que realiza a análise estática propriamente dita. Esse módulo depende diretamente da infraestrutura do compilador Roslyn e utiliza estruturas de árvore de sintaxe abstrata para percorrer o código-fonte por meio de visitantes especializados. Durante essa travessia, são coletadas informações detalhadas sobre os elementos do sistema, como classes, métodos e, principalmente, os diferentes tipos de dependência entre componentes. Os dados extraídos são então transformados em estruturas intermediárias mais acessíveis por meio de um mecanismo de mapeamento automático provido por uma biblioteca externa, permitindo desacoplar a representação interna da complexidade da AST.
Uma vez processadas, essas informações são encaminhadas ao módulo Layers, que é responsável por organizar os elementos do sistema em agrupamentos lógicos chamados camadas. Essas camadas não dependem necessariamente da estrutura física do projeto, sendo definidas com base em critérios semânticos como padrões de nomenclatura ou namespaces. O módulo Layers mantém e gerencia essas coleções de elementos, permitindo sua reutilização em múltiplas regras arquiteturais e atuando como uma camada intermediária entre os dados brutos da análise e a definição de regras.
A definição dessas camadas depende diretamente do módulo Filters, que encapsula a lógica de seleção dos elementos do sistema. Esse módulo fornece mecanismos para aplicar critérios específicos sobre os dados analisados, como filtragem por namespace, nome de classe ou outras propriedades. Os filtros são combinados por operadores lógicos, permitindo a construção de seleções complexas que servem de base para a definição das camadas. Assim, Filters e Layers operam de forma integrada para transformar os dados extraídos em representações arquiteturais significativas.
Após a definição das camadas, o fluxo segue para o módulo Conditions, que representa a etapa final do processamento. Esse módulo é responsável por avaliar as regras arquiteturais sobre as camadas previamente definidas. Ele contém implementações específicas para diferentes tipos de verificação, incluindo regras de dependência, ausência de dependências obrigatórias e validações customizadas. Para cada regra definida, o sistema percorre os elementos das camadas envolvidas e verifica se as condições estabelecidas são satisfeitas, identificando violações quando ocorrem discrepâncias. O módulo também suporta a execução de regras customizadas definidas pelo usuário, nas quais uma interface específica permite injetar lógica adicional baseada nos dados analisados.
A comunicação entre os módulos segue um fluxo unidirecional e encadeado, caracterizando um pipeline de processamento. O módulo Arch inicia o processo e depende do módulo AST para obter os dados estruturais do código. Esses dados são então organizados pelo módulo Layers com auxílio do módulo Filters. Por fim, o módulo Conditions consome essas estruturas para aplicar as regras e produzir os resultados. Não há evidência de comunicação bidirecional significativa entre os módulos, o que reforça um estilo arquitetural em camadas com separação clara de responsabilidades.
A API exposta ao usuário segue o padrão de projeto Builder combinado com uma interface fluente, permitindo que regras arquiteturais sejam construídas por meio de encadeamento de chamadas. Internamente, esse processo resulta na construção de objetos que representam regras completas, contendo tanto os filtros quanto as condições associadas. Quando a execução é acionada, essas regras utilizam os dados previamente carregados para evitar reprocessamento, indicando uma estratégia de pré-carregamento e reutilização de dados em memória.
O sistema também interage com componentes externos, principalmente o compilador Roslyn, que fornece a capacidade de análise sintática e semântica do código, e a biblioteca de mapeamento de dados, utilizada para transformar estruturas internas. Além disso, há integração indireta com frameworks de teste, permitindo que as verificações arquiteturais sejam executadas em conjunto com testes automatizados, embora essa integração ocorra fora do núcleo arquitetural da ferramenta.
Do ponto de vista de padrões arquiteturais, o sistema combina uma arquitetura em camadas com um pipeline de processamento e utiliza o padrão Builder para construção das regras, além de empregar o padrão Visitor no módulo de análise da AST. A separação entre análise, modelagem de camadas e verificação de regras evidencia uma forte modularização, permitindo extensibilidade, especialmente na definição de novas regras e critérios de análise.
O fluxo de dados inicia com a leitura da solução, passa pela extração e transformação de তথ্য estruturais do código, segue pela organização em camadas lógicas e culmina na avaliação de regras, resultando em um conjunto de violações arquiteturais que podem incluir informações detalhadas como localização no código. Esse fluxo é executado de forma sequencial e otimizada, com carregamento único dos dados e múltiplas avaliações sobre a mesma base, garantindo eficiência e consistência nas verificações.
